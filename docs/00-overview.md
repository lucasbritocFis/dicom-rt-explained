
# DICOM-RT: An Overview

## What you'll learn in this chapter

- What DICOM-RT is and why it exists as a separate "extension" of DICOM
- The five (six, including ion plans) core objects of the framework
- How they reference each other to form a coherent treatment record
- Where to look when something is missing or broken

## 1. The problem DICOM-RT solves

DICOM (Digital Imaging and Communications in Medicine) was designed in 1985 to standardize how medical images are exchanged between modalities — primarily CT, MR, US, and X-ray scanners. It worked. By the 1990s, virtually every imaging device on Earth spoke DICOM.

Radiotherapy posed a different problem. A CT scan is a single artifact: a stack of images. A radiotherapy treatment is not. It is a **process** that involves:

- An imaging study (CT, typically)
- **Contours** drawn over that imaging — where the tumor is, where the organs to avoid are
- A **plan**: which beams, in which directions, at what energy, modulated how
- A **calculated dose distribution** — the 3D map of how much radiation each voxel of the patient will receive
- **Imaging at treatment time** — verifying the patient is positioned correctly
- A **record** of what was actually delivered during each daily fraction

These are different types of data, generated by different software, exchanged between different systems (treatment planning systems, oncology information systems, linear accelerators, record-and-verify systems). They cross-reference each other intricately: a plan refers to a structure set, which refers to an imaging study, which is acquired on a specific machine.

DICOM-RT is the **set of standardized object types and their cross-references** that makes this exchangeable.

## 2. The objects

DICOM-RT defines a handful of distinct object types ("IODs" in DICOM jargon — Information Object Definitions). Each is a separate file when serialized. Each has a specific role.

| Object | Modality tag | Role | Typical size |
|---|---|---|---|
| **RT Image** | `RTIMAGE` | A 2D image acquired at treatment time (portal image, DRR, kV image) | ~1-5 MB |
| **RT Structure Set** | `RTSTRUCT` | The contours (ROI) drawn over a 3D imaging study | ~0.5-5 MB |
| **RT Plan** | `RTPLAN` | The prescription: beams, geometry, modulation, MUs | ~0.1-2 MB |
| **RT Dose** | `RTDOSE` | The calculated 3D dose distribution | ~10-200 MB |
| **RT Treatment Record** | `RTRECORD` | What was actually delivered during a fraction | ~0.05-0.5 MB |
| **RT Ion Plan** | `RTIONPLAN` | Variant of RT Plan for proton / heavy ion therapy | ~0.1-5 MB |

A complete external-beam treatment course on a clinical TPS will produce, at minimum: an imaging study (CT, technically not RT but referenced), one RT Structure Set, one RT Plan, and one RT Dose. Multiple plans (boost, adaptive replans) produce multiple sets.

## 3. How they reference each other

This is the part that confuses everyone the first time.

DICOM uses universal identifiers (UIDs) — long numeric strings like `1.2.840.10008.5.1.4.1.1.481.5` — to reference one object from another. Each object instance has its own UID. To say "this plan was made for that structure set," the RT Plan stores the UID of the RT Structure Set inside one of its tags.

The reference chain (simplified):

```
   CT Image Series
        ↑  (referenced by Frame of Reference UID)
   RT Structure Set
        ↑  (referenced by Referenced Structure Set Sequence)
   RT Plan
        ↑  (referenced by Referenced RT Plan Sequence)
   RT Dose
   RT Treatment Record
```

Three pieces of glue you should know by name:

- **Frame of Reference UID** — anchors all objects that share the same geometric coordinate system. Two objects with the same Frame of Reference UID are "in the same world" — their (x, y, z) coordinates are comparable. Different Frame of Reference UIDs mean different worlds.
- **Series Instance UID** — identifies one specific imaging series.
- **SOP Instance UID** — identifies one specific object instance (one RT Plan, one RT Dose, etc.).

When DICOM imports fail in TPS, **broken references are the #1 cause**. The RT Plan says "I belong to structure set X" but X is not in the import, or X has a different Frame of Reference UID, or X's UID was modified during an anonymization pass.

## 4. Reading order: where to start when given a folder of DICOMs

If you are handed an unknown DICOM folder representing a treatment course, the productive order to inspect it is:

1. **CT series first** — establishes the Frame of Reference and patient orientation
2. **RT Structure Set** — tells you which structures were contoured
3. **RT Plan** — tells you what was prescribed and how it would be delivered
4. **RT Dose** — tells you what dose the planning system computed would result
5. **RT Treatment Record(s)** — tells you what was actually delivered (if present)

In practice, most introductory tasks (extracting a prescription, listing OARs, computing DVH) only need 2-3 of these objects.

## 5. Common pitfalls

- **Compressed transfer syntaxes** — some scanners and PACS systems emit DICOMs in JPEG Lossless or similar compressed transfer syntaxes. Many TPS reject these on import. Convert to Implicit VR Little Endian first. (See [eclipse-dicom-fix](https://github.com/lucasbritocFis/eclipse-dicom-fix) for one solution.)
- **Anonymization breaks references** — naive anonymization tools regenerate UIDs without preserving the cross-references. The plan ends up "pointing to" a structure set whose UID no longer exists.
- **Multiple Frame of Reference UIDs** in the same folder — typically means the data came from different scans (e.g., planning CT + diagnostic MR). Each TPS has its own rules about how to handle this.
- **Empty optional fields** — many DICOM-RT tags are optional. Code that assumes they exist will crash on real-world data. Use `getattr(ds, "TagName", default)` defensively.
